## TL;DR

The **core is in good shape** — the 0.9.9 process closed everything actionable and
the tooling pass on 1.1.1 is fully green. The exposure is in the **GUI**: both High
findings from the gui-1.1.0 register (arbitrary-path file-write IPC, disabled CSP)
are **still open in the current tree**, along with the Mediums. Three core changes
since v1.0.0 (Touch ID keyslot, configurable re-auth cap, local judge providers)
have not yet been seen by any independent reviewer.

## Core — no regressions; three unreviewed surfaces since v1.0.0

The 26 commits since `v1.0.0` include three security-relevant core changes that
postdate the 0.9.9 review:

1. **Touch ID master keyslot** (`41237ea`, `src/core/touchid.rs`). A random KEK in
   the login keychain wraps the master key in `master.touchid.enc` — same keyslot
   shape as YubiKey/recovery, OR-not-2FA. The boundary is honest and documented in
   both the module docs and `docs/security.md`: the biometric check is enforced
   **in-process** because a cargo-installed CLI cannot obtain the Apple entitlement
   for a true biometry-bound keychain ACL (`errSecMissingEntitlement` probed).
   Consistent with the same-UID cooperative model — but it is a **new master
   keyslot**, exactly the class of change the review process says triggers a
   re-review.
2. **Configurable re-auth cap** (`224329f`, `src/core/session.rs`). The "6h hard
   cap" that all three 0.9.9 reviewers verified as a positive property is now a
   knob: `lock.max_unlocked_secs`, clamped 15 min – 7 days, default unchanged at
   6 h. The design is sound — the cap in force is **stamped into each session file
   at unlock**, so an existing session cannot be extended after the fact, and the
   knob lives in the encrypted keyring. But a user can now configure a 7-day
   unlocked window, which is a posture change reviewers have not seen. Documented
   in `docs/security.md`.
3. **Local judge providers** (`d862e70`, LM Studio / Ollama). The fix was
   security-conscious: a reasoning model's thinking trace is **never parsed for the
   verdict** (a trace can mention a hypothetical allow-JSON mid-reasoning), empty
   content alongside a reasoning trace fails closed with a clear error, and
   transport errors no longer name the provider.

No regressions were observed against the 0.9.9 positive-property list.

## Process gap

The tree is 26 commits past `v1.0.0` with no newer tag — a dev line toward the
2.0.0 GUI release. The documented process gates every minor release on a full
independent review, and 2.0.0 is the largest surface addition yet (the entire
Tauri GUI plus Touch ID). The gui-1.1.0 register is maintainer self-audit only;
**no independent GUI review exists yet.**

## Recommended order of work before 2.0.0

1. **Fix G-1 + G-2 together** — they are the XSS-to-arbitrary-write chain, the one
   path by which a renderer bug becomes code execution. The gui-1.1.0 register
   already spells out the fix (backend-owned save dialogs or canonicalized allowed
   roots; strict CSP).
2. **G-3 and G-4** — step-up auth on `change_master`; drop `allow-read-text`; add
   clipboard auto-clear. Small changes, the biggest real-user wins.
3. **Commission the 2.0.0 independent review** per PROMPT.md, explicitly scoping in
   the GUI IPC surface, the Touch ID keyslot, the configurable re-auth cap, and
   local judge providers — then tag.

## A note on documentation discipline

Every new boundary checked here (Touch ID's in-process enforcement, the re-auth
cap clamp, session-file cap stamping) was accurately described in
`docs/security.md` before this check ran. The gap is not honesty — it is that the
known GUI fixes have not landed yet.
