# Security Review: Svault v0.7.0

**Project**: Svault (Soluzy/Svault)  
**Version reviewed**: 0.7.0  
**Review date**: 2026-05-30  
**Reviewed by**: opencode (Qwen3-Coder-Next)  
**Review location**: Local clone at `/tmp/svault-review`

---

## Executive Summary

**Overall Risk Rating for Corporate/Medium Company Use: Medium-High (Unix + daemon) → High (Windows / regulated environments)**

Svault v0.7.0 represents a significant security hardening release, particularly with the Unix daemon (0.5.0+) and the client-side key derivation (0.7.0 #3). The daemon architecture—keys held in memory, served over a `0600` Unix socket with peer-UID validation and connection limits—is a substantial improvement over the file-based session approach.

However, the **policy engine remains advisory, not enforced**. The daemon's `Get` path runs no policy/audit check; caller/scope/reason on `svault get` are self-asserted by the client, and `svault.policy.yaml` is unsigned and located by upward directory search. This undermines the "AI-aware" claim for server/CI use, as a same-UID process can bypass the policy layer entirely. The same-UID trust model is clearly documented but may be insufficient for high-security infrastructure.

The daemon makes Svault **suitable for Unix servers and CI/CD pipelines** where the threat model is "trusted same-UID processes only." It is **not suitable for Windows environments** (no daemon, no DPAPI-equivalent ACLs) or **high-security production infrastructure** without additional controls.

## The Unix Daemon (0.5.0+) — Detailed Analysis

The daemon is the most significant security improvement in v0.7.0. It replaces the file-based `.session` with an in-memory key cache served over a Unix socket.

### Architecture

- **One daemon per project** (`.svault/`)
- **Socket**: `.svault/daemon.sock` (mode `0600`, bound under tight umask)
- **PID file**: `.svault/daemon.pid`
- **Log**: `.svault/daemon.log` (rotated at ~5MB)
- **Protocol**: Newline-delimited JSON (`Ping`, `Status`, `Unlock`, `Lock`, `LockAll`, `Get`, `Shutdown`)

### Security Benefits

1. **Keys never touch disk** — The derived key (32 bytes) is held in memory only. No `.session` file is written while the daemon is running.

2. **Client-side passphrase validation** — The client (`client.rs:38-57`) opens the vault and derives the key locally. A wrong passphrase fails immediately and never reaches the daemon.

3. **Peer-UID bond** (`daemon.rs:370-406`) — The daemon calls `getpeereid` (macOS/BSD) or `SO_PEERCRED` (Linux) and refuses any peer whose UID ≠ its own. This is a strong defense-in-depth over the `0600` socket permissions.

4. **Connection limits** (`daemon.rs:415-456`) — `max_connections` (configurable, default 512) caps live handler threads. `ConnGuard` (`daemon.rs:112-117`) ensures slots are freed even on panic.

5. **Per-connection read timeout** (`daemon.rs:108`) — 30-second timeout prevents a stalled client from pinning a handler thread indefinitely.

6. **Graceful shutdown** (`daemon.rs:337-363`) — SIGTERM/SIGINT handlers flip an atomic flag. A watcher thread sets `shutdown` and connects to the socket to unblock the accept loop. Keys are zeroized and files cleaned up.

7. **Auto-lock timers** (`daemon.rs:83-85`, `daemon.rs:324-335`) — Two timers: `idle_timeout_secs` (default 900s/15min) and `max_unlocked_secs` (default 28800s/8h). A background ticker evicts expired keys.

### Remaining Risks

1. **Policy bypass** — The daemon's `Get` handler (`daemon.rs:223-253`) performs no policy evaluation. It returns any secret if the vault is unlocked, regardless of caller, scope, reason, or tier.

2. **Same-UID trust** — A malicious process running as the same UID can directly connect to the socket and retrieve secrets without policy checks. This is documented in the threat model but may be insufficient for high-security environments.

3. **No AAD in GCM** — The salt is not authenticated as part of the GCM tag (crypto.rs:47-61). While the salt is stored in plaintext, including it as AAD would provide stronger integrity.

4. **No policy file signing** — `svault.policy.yaml` is located by upward directory search with no cryptographic signature. An attacker who can write to a parent directory can inject malicious rules.

### Operational Implications

- **Unix (Linux/macOS)**: The daemon is the preferred mode. It provides strong memory safety (keys never at rest) and connection management. Suitable for CI/CD pipelines and servers where the threat model is "trusted same-UID processes."

- **Windows/non-Unix**: The daemon is stubbed out. Everything falls back to the file-based session (`.session`), which caches the derived key (not the passphrase) with `icacls` ACLs (best-effort). Less secure than the Unix daemon.

## Infrastructure, Server & CI/CD Deployment Considerations

### Current Limitations

1. **Policy not enforced** — The daemon's `Get` path bypasses all policy checks. This means `svault get`'s caller/scope/reason are self-asserted and can be bypassed by any same-UID process.

2. **Same-UID trust** — The daemon trusts any process running as the same UID. In a shared CI environment with multiple agents, this may be insufficient.

3. **Windows ACLs** — The `icacls` ACL is best-effort and can fail silently. Windows deployments are less secure than Unix.

### Improvements Due to Daemon

1. **Keys never at rest** — On Unix with the daemon, keys are held in memory only. No `.session` file is written.

2. **Auto-lock** — Idle and hard-max timers ensure keys are evicted automatically, reducing the window of exposure.

3. **Connection limits** — Prevents unbounded thread spawning from a runaway or malicious same-UID process.

4. **Graceful shutdown** — Keys are zeroized on SIGTERM/SIGINT, preventing key leakage on process termination.

### Recommended Safe Usage Patterns on Unix Servers and CI Today

1. **Use the daemon** — Always run `svault daemon start` in CI/CD pipelines and servers. This provides the strongest security model.

2. **Set strict connection limits** — On small or shared hosts, lower `daemon.max_connections` to prevent resource exhaustion.

3. **Use short auto-lock timers** — Set `lock.idle_timeout_secs` to a low value (e.g., 300s/5min) for CI/CD pipelines.

4. **Run as dedicated UID** — Use a dedicated service account for CI/CD agents to limit the same-UID trust surface.

5. **Monitor audit logs** — Review `audit.log` for denied requests or unusual patterns.

### Risks on Windows and Non-Unix Environments

1. **No daemon** — The daemon is Unix-only. Windows falls back to the file-based session.

2. **Weaker at-rest protection** — The `.session` file uses `icacls` ACLs, which are best-effort and can fail silently.

3. **Key at rest** — The derived key is cached on disk in `.session`, which is key-equivalent.

4. **No peer-UID validation** — Windows does not implement `getpeereid`/`SO_PEERCRED`, so the peer-UID bond does not apply.

### Recommendations for High-Value Secrets

1. **Do not use `svault get` for high-tier secrets** — `high`-tier secrets are denied for agents, but the daemon's `Get` path bypasses this. Use `svault secret get` for high-tier secrets.

2. **Use the daemon on Unix** — The daemon's in-memory key cache is significantly more secure than the file-based session.

3. **Consider external HSM/KMS** — For truly high-value secrets, consider integrating with an external HSM or KMS rather than relying solely on Svault.

4. **Implement additional audit controls** — Since Svault's audit log is not tamper-evident, consider shipping logs to an external SIEM with integrity guarantees.

## Recommendations for Corporate Adoption

### Prioritized, Actionable Recommendations

1. **HIGH: Enforce policy in the daemon** — The daemon's `Get` path should call the policy engine before returning any secret. This is the most significant gap in the current implementation.

2. **HIGH: Sign the policy file** — `svault.policy.yaml` should be signed with the vault key or a separate signing key. The daemon should verify the signature before applying policy rules.

3. **HIGH: Tighten same-UID trust model** — Consider requiring additional authentication for daemon connections (e.g., a per-connection token) or limiting daemon access to specific agent UIDs.

4. **MEDIUM: Improve Windows ACL robustness** — Implement DPAPI encryption for the `.session` file on Windows, or refuse to cache keys on Windows until DPAPI is available.

5. **MEDIUM: Add AAD to GCM** — Include the salt as additional authenticated data in AES-GCM to provide stronger integrity guarantees.

6. **MEDIUM: Add checksum to recovery code** — Add a checksum or parity digit to recovery codes to catch typos early.

7. **LOW: Implement per-secret salts** — Use a unique salt per secret rather than per vault for better isolation.

8. **LOW: Add HMAC to audit log** — Sign the audit log with the vault key to provide tamper-evidence.

### Should Policy Enforcement Block a 1.0.0 "Stable" Label?

**Yes.** The policy engine is a core feature of Svault's "AI-aware" claim. Without enforcement, the policy engine is advisory only and can be bypassed by any same-UID process. A 1.0.0 release should include:

- Policy enforcement in the daemon's `Get` path
- Policy file signing with the vault key
- Verification of policy file signature by the daemon

Without these, Svault is suitable for individual/low-security use but not for corporate/CI/CD infrastructure where the "AI-aware" policy enforcement is expected.

## Overall Risk Assessment by Context

| Context | Risk Rating | Justification |
|---------|-------------|---------------|
| **Personal use (Unix + daemon)** | **Low** | The daemon provides strong memory safety, peer-UID validation, and auto-lock. Policy is advisory but not critical for personal use. |
| **Personal use (Windows)** | **Medium** | No daemon, file-based session with best-effort `icacls` ACLs. Key is at rest but never the passphrase. |
| **Small team (Unix + daemon)** | **Medium** | Same-UID trust model may be insufficient if team members run malicious processes. Policy is advisory. |
| **Corporate CI/CD with AI agents on Unix** | **Medium-High** | Daemon provides strong memory safety, but policy is not enforced. A compromised agent can bypass policy and retrieve secrets. |
| **High-security production infrastructure** | **High** | Policy not enforced, same-UID trust, no AAD in GCM. Not suitable without additional controls. |
| **Mixed Windows environments** | **High** | No daemon on Windows, weaker ACLs, key at rest in `.session`. Inconsistent security model across platforms. |
| **Multi-tenant shared servers** | **Critical** | Same-UID trust model is completely inadequate. Any process on the server can connect to the daemon and retrieve secrets. |
