# Svault 0.5.0 — Consolidated Findings Register

This is the **document of record** for the 0.5.0 security review. It de-duplicates the
findings from all five independent reviews into one list, shows which models raised each
one (cross-model agreement is the strongest prioritization signal), and records our
**disposition and justification** for each. The full reviews live in
[`../reviews/`](../reviews/).

- **Version:** 0.5.0
- **Review date:** 2026-05-29
- **Reviewers (5):** Grok 4.3 (xAI), GLM-5-1 (z-ai), Gemini 3.5 Flash, DeepSeek-V4-Pro, Claude Opus 4.8 (Anthropic)
- **Tooling pass:** `cargo test --locked` (74 pass), `cargo clippy -D warnings` (clean), `cargo audit` (0 vulns, 2 transitive advisories — see #11)
- **Dispositions decided & signed off by:** Nima Karimi (maintainer) — 2026-05-29

### Legend

**Severity** — highest rating any reviewer assigned.

**Status:**
- **Approved** — reviewed and signed off; the current behavior is intentional and accepted.
- **OK** — accepted, no action needed for now.
- **Planned** — accepted as real and scheduled for a future release (see [roadmap](../../roadmap.md)).
- **To fix** — will be fixed.
- **To check** — needs investigation; decision pending.

> The **Status** and **Disposition** columns are the maintainer's decisions for 0.5.0,
> grounded in the five reviews and the documented [threat model](../../security.md). The
> **Notes** column holds the maintainer's own working notes.

## Disposition & justification

| # | Status | Disposition & justification | Notes |
|---|--------|-----------------------------|-------|
| 1 | Approved | Same-UID trust is the documented threat model ([security.md](../../security.md)): Svault protects secrets at rest and gives cooperating agents an audit trail — it is not a sandbox against a hostile same-UID process. The `SO_PEERCRED`/`getpeereid` peer-UID bond is the accepted approach. | Peer-UID bond — OK. |
| 2 | Planned | Policy engine work is planned — evaluate moving policy evaluation + audit into the daemon so the socket becomes the enforced choke point, and state plainly that the policy layer is an honesty/audit layer today. | Policy engine planned. |
| 3 | Planned | Socket improvement planned — derive the key client-side and send only the derived key / a session token, so the passphrase never crosses the socket. | Socket improvement. |
| 4 | To fix | Update the TUI so the daemon is always preferred; deprecate `.session` on Unix. On Windows, add DPAPI-backed protection or refuse-to-cache (require re-entry). Until then, document loudly that `.session` = cleartext passphrase at rest. | Update TUI — daemon always preferred. |
| 5 | Planned | Policy config work planned — sign the policy file (or pin it to the vault) and stop the unbounded upward search / restrict to a trusted root. | Policy config. |
| 6 | To check | Investigate extending zeroization to the passphrase `String` and returned secret values (`Zeroizing`/`secrecy`); derived keys are already `ZeroizeOnDrop`. | TBD — to check. |
| 7 | OK | Random 96-bit nonce per write makes key+nonce reuse risk negligible, and re-key rotates the salt. Per-write salt rotation can be done as cleanup; not a blocker. | OK. |
| 8 | Planned | Add a max-connections ceiling + per-connection timeout, and include performance / max-out tests across many threads. | Planned — perf + max-out multi-thread tests. |
| 9 | Planned | Add a `cargo audit` / `cargo-deny` gate to CI. Schema/config still to be defined. | Planned — TBD, define schema. |
| 10 | Planned | TUI-only transitive advisories, not in the crypto/daemon path; a `ratatui` bump clears both. Will be fixed. | Planned — will be fixed. |
| 11 | To check | Verify what's feasible for release-artifact integrity — SHA-256 checksums first, then evaluate signing/SBOM/provenance. | To check — to verify. |
| 12 | To check | Create flow already warns + confirms. Investigate an entropy-based estimate with an enforced floor + `--force` for non-interactive use. | To check. |
| 13 | To fix | Replace bare `lock().unwrap()` with `lock().unwrap_or_else(PoisonError::into_inner)` so a panicking handler can't kill the daemon without zeroizing. Check zeroization on that path. | To fix — check zeroizing. |
| 14 | Planned | Create `recovery.enc` / export bundles `0600`, make the export gitignore guard non-optional, consider a distinct export passphrase. Cover import/export hardening more fully later. | Planned — cover in/out more later. |
| 15 | OK | Needs improvement, OK for now: local logs are advisory and never contain secret values. For real assurance, ship them to an append-only/remote sink; document that local logs are not tamper-evident. | Needs improvement; OK for now. |
| 16 | Planned | Create `.svault/` as `0700` and set the socket mode atomically (umask around `bind`, or bind in a private temp dir). Also explore binding a machine ID at vault creation so the vault is stuck to the hardware it was created on — check options. | Planned — also: machine-ID bind to hardware at vault creation; check options. |
| 17 | Planned | Daemon hygiene cleanup — `SIGTERM` handler that zeroizes, log rotation/size cap, tighten the start race, tune the ticker interval. | Planned — Rust/runtime hygiene. |
| 18 | OK | 160-bit code is strong; a checksum digit is a UX nicety to catch typos before Argon2id. Backlog. | OK. |
| 19 | OK | Minor perf/exposure nit; a separate HMAC/known-payload verification step would avoid decrypting all secrets just to check the passphrase. Backlog. | OK. |
| 20 | To fix | Replace the `unwrap()` on the length slice (`vault.rs:152`) with a checked length guard returning an error. Availability-only, same-user. | To fix — Rust things. |
| 21 | OK | Salt change yields the wrong key (auth failure) and `meta.yaml` is separately HMAC-protected, so impact is minimal. Bind salt/context as AAD as cleanup. | OK. |
| 22 | Planned | Config-related: verify `meta.yaml` (HMAC) on the `get`/fallback path too, so the advertised metadata-integrity guarantee covers the policy gate. | Planned — config related. |
