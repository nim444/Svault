## Headline: all three reviews independently confirm the known gap

The single most important finding is unanimous across all three reviewers and
already tracked here as **#2/#5/#22**: **the policy engine is advisory, not
enforced.** The daemon `Get` path runs no policy or audit check, caller/scope/
reason are self-asserted, and `svault.policy.yaml` is unsigned and located by an
upward directory walk. A same-UID process bypasses the entire policy layer by
calling `svault secret get`, by speaking the socket protocol directly, or by
reading the `.session` key. **All three reviews explicitly state this should
block a 1.0.0 "stable / enforced policy" label.** The reviews validate the
priority and sharpen the requirements (N-1, N-2, N-5 fold into it). Per maintainer
decision this policy work is **deferred to a dedicated later release** — 0.8.0 is
a hardening release that acts on everything *else* the review surfaced (see the
"0.8.0 scope" section below). Policy enforcement is deferred, not dropped, and
remains the gate for 1.0.0.

The same-UID trust model itself was judged **reasonable and clearly
communicated** — the issue is not the boundary, it is that the headline feature
is bypassable *within* it.

## 0.8.0 scope (maintainer decision)

**0.8.0 is a security-review-response + hardening release. The policy engine is
deliberately deferred to a later release — it is NOT dropped.** It remains the
single substantive gap before a 1.0.0 "enforced policy / stable" label, exactly
as all three reviews flagged. Splitting it out keeps 0.8.0 a clean, shippable
hardening release rather than blocking on the largest piece of work.

**Landed in 0.8.0:**
- Owner-only TUI export (N-3), `0700` import dir (N-4 + regression test), `0600` `daemon.log` after rotation (N-10).
- Daemon transport zeroization — serialized reply buffer + in-memory `Response::Secret` wiped after write (N-6).
- `libc::sigaction` instead of `signal()` (N-9).

**Deferred to a later release (the policy thread — gates 1.0.0):**
- Move policy evaluation + audit into the daemon `Get` path (#2, N-5).
- Authenticate the caller out-of-process via peer creds / per-agent token (N-1).
- Sign/pin `svault.policy.yaml` and anchor discovery to the vault (#5).
- **Fail closed** on an unparseable policy (N-2).
- Verify `meta.yaml` on the fallback `get` path (#22).

**Accepted / backlog (not 1.0.0 blockers):** Windows atomic DACL (N-7),
tamper-evident audit sink (N-8/#15), `Unlock` rate-limit (N-11), tunable Argon2id
(N-12), dictionary-aware entropy (N-13), `load_secrets` map residual (N-6 part).
