First-run onboarding

A linear, can't-skip-the-important-bits flow. Honest disclaimer → one master passphrase → one-time recovery code → optional hardware key. Lands the user in an empty vault list, ready to create their first vault.

4 steps, shown left → right. Same window, content swaps. Stepper always visible so the user knows where they are.

Sign in & sign out — app-level auth

The returning-user gate. The app opens locked: no vault is readable in the GUI until the master passphrase (or an enrolled YubiKey) unlocks the session. Signing out only ends the GUI session — the daemon, MCP server, and vault lock state are left untouched and are managed from the dashboard. A 6-hour re-auth cap applies to every unlock path.

MCP — the agent door

The supported way agents reach secrets: a local stdio JSON-RPC server (svault mcp). It never sees the passphrase, serves only unlocked state, and runs every request through the same policy + judge gate. This screen is status + wiring, not a new trust model.
